# 猫町OSIROイベント設定チェックCLI 外部設計書

## 1. 目的

本システムは、猫町倶楽部のOSIRO管理画面に登録されている募集中イベントを自動確認し、イベント設定の不備をSlackへ通知するCLIアプリケーションである。

確認対象は主にイベントチケットの設定であり、チケット名、金額、販売対象者、オンライン参加URL、主催者からのお知らせ、プラン変更チケットの有無をチェックする。

## 2. 利用者

- イベント設定を確認する運用担当者
- OSIRO管理画面にログインできる管理者
- Slack通知を受け取って修正対応する担当者

## 3. 実行方式

### 3.1 手動実行

プロジェクト直下の以下を実行する。

```powershell
.\run-check.bat
```

またはNode.jsコマンドで直接実行する。

```powershell
npm run check
```

### 3.2 定期実行

Windowsタスクスケジューラーで毎日0時に実行する。

登録済みタスク名:

```text
Nekomachi OSIRO Event Check
```

実行内容:

```text
powershell.exe -NoProfile -ExecutionPolicy Bypass -File "C:\asuka-windows\app\イベントページ自動チェック\scripts\run-check.ps1"
```

### 3.3 初回ログイン

OSIRO管理画面へのログイン状態を保存するため、初回またはログイン切れ時に以下を実行する。

```powershell
npm run auth
```

ログイン後、`storageState.json` にブラウザのログイン状態が保存される。

## 4. 入力

### 4.1 OSIRO管理画面

イベント一覧URL:

```text
https://nekomachi-club.com/admin/events?state=yet_end
```

取得する主な情報:

- イベント名
- 詳細URL
- 開始日時
- 終了日時
- 会場
- チケット名
- チケット金額
- 販売対象者
- オンライン開催有無
- オンライン参加URL
- 主催者からのお知らせ

### 4.2 環境変数

`.env` に以下を設定する。

```env
SLACK_BOT_TOKEN=xoxb-xxxxxxxxxxxxxxxx
SLACK_CHANNEL_ID=C0BCXMXG745
HEADLESS=true
```

### 4.3 ルール設定

`config/rules.yaml` にオンライン・オフラインごとの基本チケットルールを定義する。

販売対象者タグ:

```text
オン: オンライン会員
オフ: 地域会員
ハイ: ハイブリッド会員
外: 非会員
A / U-22 / B: 旧会員
```

## 5. 出力

### 5.1 コンソール出力

チェック結果のサマリーとNG詳細を標準出力に表示する。

### 5.2 Slack通知

Slackに以下を通知する。

- チェック対象件数
- 対象外件数
- OK件数
- NG件数
- 実行日時
- NGイベント一覧
- イベント名と開催日時
- イベント種別
- 詳細URL
- 不備内容

NGイベントは開催日時の降順で表示する。開催日時を取得できないイベントは末尾に表示する。

### 5.3 ログファイル

`run-check.bat` またはタスクスケジューラー経由で実行した場合、`logs/` にログを保存する。

ファイル名:

```text
logs/check-YYYYMMDD-HHMMSS.log
```

### 5.4 障害調査用ファイル

詳細取得に失敗した場合、可能な範囲で以下を保存する。

```text
artifacts/screenshots/
artifacts/html/
```

## 6. チェック対象イベント

OSIRO管理画面の募集中イベント一覧から、詳細URLが以下形式のものを対象にする。

```text
/admin_events/{id}/edit
```

以下の操作リンクはイベント名として扱わない。

- 削除する
- 非公開にする
- 複製する
- 編集する
- 参加者
- 分析
- チケット

## 7. イベント種別判定

### 7.1 対象外

イベント名に以下を含む場合は対象外とする。

- 予告
- 一覧
- 事務局決済

### 7.2 オフライン

イベント名に以下を含む場合はオフラインイベントとする。

- 【東京】
- 【大阪】
- 【京都】
- 【福岡】
- 【名古屋】
- 東京開催

`〖東京〗` のような括弧表記は `【東京】` と同等に扱う。

### 7.3 オンライン

対象外・オフラインに該当しないイベントはオンラインイベントとする。

## 8. オンラインイベントのチェック内容

### 8.1 基本チケット

オンラインイベントでは以下のチケットを期待する。

| 会員種別 | 条件 | 金額 |
| --- | --- | ---: |
| オンライン会員 | 1回目 | 0円 |
| オンライン会員 | 2回目以降 | 800円 |
| 地域会員 | 通常 | 800円 |
| ハイブリッド会員 | 通常 | 0円 |
| 非会員 | 通常・初参加 | 1,100円 |

非会員チケットは、通常チケットと初参加チケットの2つが存在する場合は許容する。

### 8.2 オンライン参加URL

オンライン開催がONのチケットはオンライン参加URLが入力されている必要がある。

また、通常チケット間のオンライン参加URLは一致している必要がある。

### 8.3 主催者からのお知らせ

通常チケット間で主催者からのお知らせが一致している必要がある。

締切時刻はイベント開始5分前である必要がある。

例:

```text
18:30開始 -> 18:25までに
```

## 9. オフラインイベントのチェック内容

### 9.1 基本チケット

オフラインイベントでは以下のチケットを期待する。

| 会員種別 | 条件 | 金額 |
| --- | --- | ---: |
| 地域会員 | 1回目 | 0円 |
| ハイブリッド会員 | 1回目 | 0円 |
| 地域会員 | 2回目以降 | 1,800円 |
| ハイブリッド会員 | 2回目以降 | 1,800円 |
| オンライン会員 | 通常 | 1,800円 |
| 非会員 | 通常・初参加 | 2,300円 |

### 9.2 参加種別

オフライン読書会では以下の2種類のチケットが必要である。

- 読書会のみ参加
- 懇親会まで参加

同一会員種別でこの2種類が存在する場合、重複として扱わない。

### 9.3 ゲストイベント

イベント名またはチケット構成からゲストイベントと判断できる場合は、以下の金額でチェックする。

| 会員種別 | 条件 | 金額 |
| --- | --- | ---: |
| 地域会員・ハイブリッド会員 | 1回目 | 800円 |
| 地域会員・ハイブリッド会員 | 2回目以降 | 3,000円 |
| オンライン会員 | 通常 | 3,000円 |
| 非会員 | 通常・初参加 | 3,500円 |

## 10. 共通チェック

### 10.1 プラン変更チケット

複数チケットのイベントでは、旧会員向けのプラン変更チケットが必要である。

プラン変更チケットとして認識する表記:

- プラン変更後にお申し込み下さい。
- プラン変更後にお申込みください。
- 新プラン切り替え後にお申し込み下さい。
- プラン切り替え後にお申し込み下さい。

販売対象者には以下が含まれている必要がある。

- A
- U-22
- B

また、A / U-22 / B はプラン変更チケット以外に含めない。

### 10.2 1チケットのみのイベント

チケットが1つだけのイベントは、無料であればOKとする。

この場合、プラン変更チケット、主催者からのお知らせ、オンラインURLなどの通常チェックは行わない。

### 10.3 固定費チケット

チケットが2つで、片方がプラン変更チケットの場合、もう片方を固定費チケットとして扱う。

固定費チケットは金額チェックを行わないが、販売対象者に以下が含まれている必要がある。

- オン
- オフ
- ハイ
- 外

### 10.4 特殊申込済みチケット

チケット名に以下を含む場合、重複チェックと金額チェックから除外する。

```text
お申し込みいただいた方
お申込みいただいた方
```

`読書会なし` と `読書会セット` は本のタイトルとして扱わない。

## 11. エラー表示

チケット単位のエラーには、上から何番目のチケットかを表示する。

例:

```text
[3番目] チケット「地域会員」: 金額が期待値と異なります。期待: 800円 / 実際: 1100
```

主催者からのお知らせが1チケットだけ異なる場合は、該当チケット名と位置を表示する。

## 12. 運用上の注意

- Slack投稿先チャンネルにはBotを招待しておく。
- `storageState.json` はログイン情報を含むためGit管理しない。
- OSIRO管理画面のHTML構造が変わると、詳細取得に失敗する可能性がある。
- ログイン期限が切れた場合は `npm run auth` を再実行する。
